## Use case

Mirror the legacy `removeIce()` editor action from
`public/modules/ui/ice-editor.js`:

```js
function removeIce() {
  // ...confirm dialog...
  Ice.removeIce(+elSelected.attr("data-id"));
  // ...close editor...
}
```

User flow today: open the Edit Ice dialog → click **Remove** → the selected
glacier or iceberg disappears from the map.

The AI has no equivalent today. We add `remove_ice` so an agent can delete an
ice element by id and have the corresponding SVG element disappear.

## Exact behaviour

1. Validate the `id` argument:
   - must be a number
   - must be finite (rejects `NaN`, `Infinity`, `-Infinity`)
   - must be an integer
   - must be `>= 0` (ice ids start at 0; see `IceModule.getNextId`)
2. Look up the matching ice element in `pack.ice` **before** delegating to
   `Ice.removeIce`, so we can:
   - report the element's `type` (`"glacier" | "iceberg"`) and `cellId` in the
     happy-path response
   - return a clean "ice element not found" error rather than letting
     `Ice.removeIce` silently no-op
3. Call the runtime's `removeIce(id)` (default = `window.Ice.removeIce(id)`).
   That is the same call path as the editor: it splices out of `pack.ice` and
   then calls `redrawGlacier(id)` or `redrawIceberg(id)` based on type, which
   makes the SVG element disappear.
4. Verify the entry is no longer in `pack.ice`. If it still is, surface an
   error ("removal failed") — defends against a stub `Ice.removeIce` that
   no-ops.
5. Return `okResult({ id, type, cell_id })`.

`cell_id` matches the `cellId` field from the ice element. For glaciers
generated by `IceModule.generate`, `cellId` may be `undefined` — in that case
we report `null` so the JSON response has a stable shape.

## Runtime-injection seam

```ts
export interface RemoveIceRef {
  i: number;
  type: "glacier" | "iceberg";
  cellId: number | null;
}

export interface RemoveIceRuntime {
  findIce(id: number): RemoveIceRef | null;
  removeIce(id: number): void;
  getIceArray(): readonly { i: number }[] | null;
}

export const defaultRemoveIceRuntime: RemoveIceRuntime;

export function createRemoveIceTool(runtime?: RemoveIceRuntime): Tool;
export const removeIceTool: Tool;
```

`defaultRemoveIceRuntime`:
- `findIce(id)` reads `pack.ice` via `getPack()`, finds the entry with
  `entry.i === id`, returns `{ i, type, cellId: cellId ?? null }`. Returns
  `null` for "not found"; throws when `pack` / `pack.ice` are missing (so the
  caller can surface the error).
- `removeIce(id)` looks up `window.Ice.removeIce`; throws if unavailable; calls
  `Ice.removeIce(id)`.
- `getIceArray()` returns the current `pack.ice` array (or `null` if
  unavailable), used for the post-call presence check.

The tool's `execute` wraps each runtime call in try/catch and forwards the
message via `errorResult`.

## Tests (Vitest)

Tests live in `src/ai/tools/remove-ice.test.ts`. They cover both an
injected-runtime path (most tests) and the default runtime against
`globalThis` stubs.

1. **Happy path iceberg**: pack.ice contains `{i:7, type:"iceberg", cellId:42, ...}`.
   `remove_ice {id:7}` → result reports `{ok:true, id:7, type:"iceberg", cell_id:42}`;
   runtime `removeIce` was called with `7`; pack.ice no longer contains entry 7.
2. **Happy path glacier**: same, with `type:"glacier"` (and `cellId` may be
   `null` since the generator can omit it for glaciers).
3. **id not present** → error `"No ice element found with id 99."`; remove not called.
4. **id is float (1.5) / NaN / Infinity / non-number / null / object** → error;
   remove not called.
5. **id missing entirely** → error.
6. **pack.ice missing in default runtime**: stub `globalThis.pack = {}` →
   error mentions `pack.ice`.
7. **pack missing in default runtime**: error mentions `pack`.
8. **Ice missing in default runtime**: error mentions `Ice.removeIce`.
9. **Ice.removeIce throws** → error message forwarded; pack.ice unchanged.
10. **Ice.removeIce no-ops (entry still present)** → `"Failed to remove ice
    element …"`.
11. **Default runtime happy path**: stub `globalThis.pack = {ice: [...]}` and
    `globalThis.Ice = {removeIce: id => splice}` — confirm round-trip.
12. **Tool name + registry round-trip**: the exported `removeIceTool.name` is
    `"remove_ice"` and the registry exposes it after registration.

Tests stash and restore the existing `globalThis.pack` and `globalThis.Ice`
in `beforeEach` / `afterEach` to avoid leaking between cases.

## Self-review

Re-read plan + tasks. Notes:

1. **Why not `parseEntityRef`**: that helper enforces `value > 0` and rejects
   `0`. Ice ids start at 0 (see `IceModule.getNextId`), so a vanilla "first
   ice element generated" has `i === 0` and would be unfixable through
   `parseEntityRef`. We write a small inline validator instead. The schema
   only takes `id` (integer), no string form, so this is simpler than the
   `parseEntityRef` path anyway.

2. **`findIce` error vs null**: split it the way other tools do — return
   `null` for "entry not found" (so the tool returns a clean
   `"No ice element found with id …"` message), throw for "pack / pack.ice
   missing" (so the tool surfaces a different error). This way the test
   suite can distinguish the two cases.

3. **`getIceArray` rationale**: needed for the post-call presence check.
   `Ice.removeIce` may silently no-op if the entry is gone (its real
   implementation already does that for non-matching ids); a stub passed in
   tests can also no-op. We re-read `pack.ice` after the call to confirm
   removal — this is what the spec requires ("verify the entry is no longer
   in `pack.ice`").

4. **`cell_id` handling**: glacier entries from `IceModule.generate` do not
   set `cellId` (only icebergs do, see `ice.ts` lines 59-64 vs 87-93). So
   `cellId` may be `undefined`. We coerce to `null` in the returned ref so
   JSON serialisation is stable (`undefined` would drop the key entirely).

5. **Description string**: matches the style of `remove_burg` / `remove_river`
   — references the editor button and the underlying generator method, and
   explicitly notes that the confirm dialog is skipped.

6. **Module path correctness**: `Ice` global is registered in
   `src/modules/ice.ts` line 175 (`window.Ice = new IceModule()`), so
   `getGlobal<IceModule>("Ice")` is the correct accessor.

7. **No regression risk to lint**: new code uses only patterns already in use
   (typed `getGlobal`, kebab-case file name, double quotes, 2-space indent).

8. **Test file naming**: `remove-ice.test.ts` matches the convention in
   `src/ai/tools/`.

9. **Tasks file already enumerates the negative-id case** — covered.
